# Руководство мейнтейнера

Как устроены данные и как добавлять/оформлять тафсиры, переводы, словари.
Для пользователей — `HELP.md`. История изменений — `CHANGELOG.md`.

---

## 1. Структура данных

```
data/
  config.json            ← список всех источников (что показывать)
  config.js              ← то же, для offline (file://)
  index/<id>.json        ← индекс поиска (генерируется)
  tafsirs/
    <id>/<surah>.json    ← чанк по суре:  { "<аят>": "текст" }
    <id>.json            ← весь источник целиком (для поиска)
    <id>.js              ← bundled-версия (offline)
  qpc/v1/                ← данные печатного мусхафа (генерируются build_qpc.py)
    surah/<s>.json       ← глифы по аятам
    page/<p>.json        ← строки по страницам
    meta.json            ← суры/джузы/хизбы/страницы → позиции
fonts/                   ← самохостимые шрифты (.woff2)
  qpc-v1/p1..p604.woff2  ← постраничные шрифты печатного мусхафа
```

Каждый тафсир/перевод/словарь — это просто `{ "номер_аята": "текст" }` по сурам.
Грузится лениво (`loadChunk`), кэшируется (LRU). Можно заполнять постепенно:
сделал суру 1 — работает; завтра суру 2 — работает.

## 2. Как добавить новый источник (тафсир / перевод / словарь)

### Шаг 1. Файлы чанков
Создай папку `data/tafsirs/<id>/` и клади туда `<surah>.json`:
```json
{
  "1": "Текст к аяту 1 (можно с Markdown).",
  "2": "Текст к аяту 2."
}
```
`<id>` — латиницей без пробелов (напр. `irab`, `wbw_ru`, `my_tafsir`).

### Шаг 2. Регистрация в `data/config.json` → массив `"tafsirs"`
```json
{
  "id": "irab",
  "file": "tafsirs/irab.json",
  "name": "Иъраб",
  "nameAr": "الإعراب",
  "author": "",
  "group": "modern_ar",
  "kind": "tafsir",
  "lang": "ar",
  "era": "",
  "tags": [],
  "year": ""
}
```
Поля:
- `kind`: `"quran"` (арабский оригинал, особый рендер) · `"translation"` ·
  `"tafsir"`. Для пословного словаря удобно `"translation"` или `"tafsir"`.
- `lang`: `"ru"` или `"ar"` (влияет на направление текста: ar → RTL).
- `group`: id группы из `config.json` → `"groups"` (translations / russian /
  modern_ar / classical_ar …). Можно завести свою группу.
- `about`: `true` — если есть вступление/предисловие в `data/about/<id>.md` (см. ниже).
  Кнопка **Паспорт** в каталоге показывается у каждого источника; `about` добавляет
  свободный Markdown-блок под автоматической карточкой из `config.json`.
- `caution`: `true` — для сомнительных/нестандартных переводов. В окне «О переводе»
  сверху автоматически выводится баннер-предупреждение («только в ознакомительных
  целях, «как есть», возможны ошибки и вероучительные отклонения»). Текст баннера
  единый, задаётся в коде (`openAbout`), дублировать в `about/<id>.md` не нужно.
  В каталоге показывается метка «⚠ не проверено».
- `approved`: `true` — полностью проверен и одобрен. В каталоге метка «✓ одобрено»
  (зелёная). Ставится вручную на доверенные источники; остальные нейтральны.
- `fill`: `0..1` — доля реальных аятов (наполненность). **Не редактировать вручную** —
  пересчитывается скриптом `python3 compute_fill.py` (после добавления/правки источника).
  В каталоге у неполных (<100%) показывается зелёная полоса прогресса.
- `tags: ["draft"]` — метка «неполный» в каталоге (для частичных источников).
- Дополнительные поля паспорта (необязательные): `translator`, `editor`, `source`,
  `sourceName`, `sourceUrl`, `edition`, `license`, `licenseStatus`, `reviewStatus`,
  `reviewNote`, `version`. Они автоматически появятся в окне паспорта, если заполнены.

### Скан-источник (`kind: "image"`) — для нераспарсиваемых книг
Когда текст не вытащить (кривой OCR, две колонки, поэтич. вёрстка) — показываем
**сканы страниц**. Источник в конфиге: `kind:"image"`, `imgdir:"img/<id>"`, `fill`
ставится вручную (compute_fill его не трогает).
- Рендер страниц из PDF: `pdftoppm -png -r 150 -f <стр> -l <стр> book.pdf prefix`
  → клади как `data/img/<id>/pNNN.png`.
- Карта по сурам: `data/img/<id>/<surah>.json` = `{"text":["p016.png"],"notes":["p627.png"]}`
  (`text` — страницы перевода суры, `notes` — страницы примечаний к ней).
- Предисловие: `data/img/<id>/_about.json` = `{"images":["p013.png"]}` — показывается
  картинками в окне «О переводе» (кнопка ℹ).
- В ридере источник выводится блоком сверху суры с вкладками **«Текст»/«Примечания»**
  (`renderImageBlock`/`imgTab`), per-ayah-строки для него не строятся.
- Какая PDF-страница что содержит: `pdftotext -layout book.pdf out.txt`, затем split
  по `\f` и поиск нужных глав/разделов (см. как делалось для Богуславского).

### Шаг 2б. (если нужно) вступление «О переводе/тафсире»
Предисловие, «от переводчика», вводная статья — кладём в `data/about/<id>.md`
(Markdown, грузится лениво при клике на **ℹ**, рендерится в модальном окне; в
шапке окна автоматически показываются `name`/`nameAr`/`author`/`year` из конфига).
Кнопка **Паспорт** в каталоге появляется у всех источников. Чтобы под автоматической
карточкой показывался отдельный текст вступления, поставь `"about": true` в записи
источника и добавь файл `data/about/<id>.md`. Файл может быть любой длины (окно
прокручивается). Дублировать в offline `.js` не нужно.

Важно различать **два разных «введения»**:
- **Служебное** («о переводе», справка об источнике) → `data/about/<id>.md`, живёт в
  окне паспорта (Шаг 2б выше).
- **Авторский текст-введение самого произведения** (мукаддима тафсира, вступление к
  суре) → это часть книги, показывается в самом ридере (Шаг 2в ниже).

### Шаг 2в. (если нужно) мукаддима тафсира и вступления к сурам
Авторские введения хранятся прямо в данных источника под **зарезервированным ключом
`"0"`** — отдельного формата не требуется, схема `{сура:{аят:текст}}` не меняется.

- **Вступление к суре** (фадаиль, мекканская/мединская, причины ниспослания) —
  ключ `"0"` *внутри объекта суры*: `data["3"]["0"] = "…текст…"`. Рендерится блоком
  **«📖 Вступление к суре»** над аятом 1 в режиме «Аяты».
- **Мукаддима всей книги** (длинное вступление Куртуби и т.п.) — отдельная
  **«сура 0»**: `data["0"] = {"1":"…раздел 1…","2":"…раздел 2…"}` (разделы нумеруются
  по порядку). В конфиге источнику ставится `"muqaddima": true` → в списке сур
  появляется пункт **«§ المقدمة · Введение»**, открывающий отдельный режим со всеми
  включёнными мукаддимами.

Ключ `"0"` **не считается аятом**: `compute_fill.py`/`validate_data.py` его игнорируют,
на `fill` он не влияет. Для offline-источника (есть `<id>.js` в `TL_DATA`) введения
надо положить и в бандл — приложение читает оттуда (см. Шаг 3). Пример (демо-заглушки)
уже есть у `qurtubi`: `data/tafsirs/qurtubi.js` + `"muqaddima": true` в конфиге —
замени тексты-шаблоны на реальную мукаддиму.

### Поля паспорта источника (окно ℹ)
Кнопка **ℹ паспорт** (у каждого источника в ридере и в шапке скан-читалки) открывает
карточку. Поля берутся из записи источника в `config.json` и показываются, только если
заполнены: `author`, `translator`, `editor`, `year`, `source`/`sourceName`+`sourceUrl`,
`edition`, `license`/`licenseStatus`, `reviewStatus`/`reviewNote`, `version`. Статус-
пилюли сверху: `approved:true` → «✓ одобрено», `caution:true` → «⚠ не проверено»,
`tags:["draft"]` → «черновик», `fill<100%` → «заполнено N%».

### Шаг 3. (если нужно) bundled `.js` для offline
Рядом с `<id>.json` положи `<id>.js`:
```js
window.TL_DATA=window.TL_DATA||{};window.TL_DATA["irab"]={ ...тот же объект... };
```
И подключи `<script src="data/tafsirs/irab.js"></script>` в `index.html`
(рядом с остальными). Для онлайн-версии (http) это необязательно — хватает JSON.

> `convert.py` умеет собирать `.json` + `.js` из текстового файла (см. §5).

## 3. Разметка (Markdown)

Текст тафсиров/переводов рендерится как **Markdown**. Поддерживается:

| Синтаксис | Результат |
|---|---|
| `**жирный**` | **жирный** |
| `*курсив*` | *курсив* |
| `` `моно` `` | моноширинный |
| `# Заголовок` … `#### ` | заголовки (4 уровня) |
| `> цитата` | блок цитаты |
| `- пункт` / `* пункт` | маркированный список |
| `1. пункт` | нумерованный список |
| `---` (отдельной строкой) | горизонтальная линия |
| `\| a \| b \|` + строка `\|---\|` | таблица |
| `[[видимое::текст карточки]]` | кликабельное слово → всплывающая карточка |
| `текст[^1]` + `[^1]: …` | сноска: индекс в тексте → нумерованный список под аятом |

Обычный текст: пустая строка = новый абзац, одиночный перенос = новая строка.
HTML экранируется (вставить произвольный HTML нельзя — безопасно).

### Сноски / примечания (`[^N]`)
Для переводов с примечаниями. В тексте ставим ссылку `[^1]`, `[^2]` (рендерится
надстрочным индексом акцентным цветом). Сами примечания — **в конце текста аята**,
каждое с новой строки в виде `[^1]: текст примечания` (может быть Markdown,
многострочным — примечание тянется до следующего `[^N]:` или конца). Под аятом
они показываются нумерованным списком (приглушённо, мельче, без слова-метки);
если суммарно длинные — сворачиваются кнопкой «Развернуть». Несколько сносок на
один аят поддерживаются. Пример значения аята в JSON:
```
"7": "Веди нас прямым путём[^1], путём тех, кого Ты облагодетельствовал[^2].\n[^1]: Сырат аль-мустаким — ислам, путь Корана и Сунны.\n[^2]: Пророки, праведники, павшие за веру и благочестивые."
```
Когда примечания стабильно огромные (фактически тафсир), лучше вынести их в
**отдельную колонку** `kind:"tafsir"` (напр. `id_notes`), а не в сноски.

### Карточки (грамматика по нажатию)
`[[слово::текст]]` рендерит «слово» как кликабельное (пунктирное подчёркивание);
по нажатию открывается карточка с «текст». Внутри карточки: `||` = перенос
строки, работают `**жирный**`/`*курсив*`. Пример:
```
[[ٱلْحَمْدُ::артикль الـ + имя «хвала»||мубтада, **марфуъ**]]
```

### Таблица — точный синтаксис
```
| Слово | Перевод |
|---|---|
| ٱلْحَمْدُ | хвала |
| لِلَّهِ | Аллаху |
```
Первая строка — заголовок, вторая — обязательный разделитель `|---|---|`,
дальше строки данных.

## 4. Пословный словарь и иъраб

Арабский аят сверху **не трогаем** (в печатном виде он неделим — глифы по
странице). Словарь/иъраб — это отдельный источник (см. §2), а его текст для
каждого аята оформляем Markdown'ом.

**Пословный словарь — вариант таблицей:**
```
| Слово | Перевод |
|---|---|
| ٱلْحَمْدُ | хвала |
| لِلَّهِ | Аллаху |
| رَبِّ | Господу |
| ٱلْعَٰلَمِينَ | миров |
```

**Пословный словарь — вариант построчно (быстрее набирать):**
```
**ٱلْحَمْدُ** — хвала
**لِلَّهِ** — Аллаху
**رَبِّ** — Господу
```

**Пословный словарь с грамматическими карточками** (из грамматического корпуса):
каждое слово — кликабельное, по нажатию карточка с разбором.
```
[[ٱلْحَمْدُ::определённый артикль الـ + имя||мубтада, **марфуъ**]] — хвала
[[لِلَّهِ::предлог لِ + لفظ الجلالة||جار ومجرور]] — Аллаху
[[رَبِّ::имя, مضاف||маджрур]] — Господу
```
Каждое слово на своей строке: кликабельный арабский + перевод. По тапу —
карточка с грамматикой (`||` — перенос строки внутри карточки). Так же можно
вставлять `[[…::…]]` в ячейки таблицы.

**Иъраб:** термины — `**жирным**`, разбор — списком, грамматика — таблицей,
пояснения — `> цитатой`, термин с разбором — карточкой `[[термин::разбор]]`.
Длинные блоки свернутся автоматически (>600 символов).

## 5. Готовый текст → JSON (`convert.py`)

Если исходник в виде текста с разделителями `сура:аят`, используй `convert.py`:
```
python3 convert.py вход.txt data/tafsirs/<id>.json
```
Он создаёт и `.json`, и `.js`. Затем при необходимости разрежь на чанки:
```
python3 split.py            # tafsirs/<id>.json → tafsirs/<id>/<surah>.json
python3 build_index.py      # обновить индекс поиска index/<id>.json
```
(Точные аргументы — внутри скриптов; формат входа описан в `README.md`.)

### Правила JSON
- Только двойные кавычки `"`.
- Кавычки внутри текста экранируй: `\"`.
- Перенос строки внутри значения — `\n`.
- Запятая после каждого элемента, **кроме последнего**.

## 6. Печатный мусхаф (QPC) и шрифты

- Данные мусхафа генерирует **`build_qpc.py`** (тянет глифы/страницы/джузы/хизбы
  из API quran.com) → `data/qpc/v1/`. Перезапуск пересобирает всё.
- Постраничные шрифты `fonts/qpc-v1/p1..p604.woff2` — самохост, ленивая загрузка
  (по одному на открытую страницу).
- Реестр шрифтов-паков — объект `PACKS` в `index.html`. Чтобы добавить другой
  пак (Indopak, Warsh, QPC v2) — добавить запись в `PACKS` + его шрифты/данные.
- Обычные шрифты (`KFGQPC Uthmanic`, `Amiri Quran`, `Reem Kufi`) — в `fonts/`,
  подключены через `@font-face`.

> При массовой докачке шрифтов с jsdelivr бывает rate-limit (битые мелкие
> файлы). Проверяй сигнатуру `wOF2` (`head -c4`), а не только размер; качай
> последовательно с ретраями, зеркало — `cdn.statically.io`.

## 7. Запуск и публикация

Локально (нужен http-сервер, иначе браузер блокирует JSON):
```
python3 -m http.server 8000
# → http://localhost:8000
```

GitHub Pages: залить всю папку, включить Pages.
**Важно:** в корне должен быть пустой файл **`.nojekyll`** — иначе Jekyll
вырезает файлы и папки, начинающиеся с `_` (а арабский лежит в
`data/tafsirs/_arabic*`), и арабский не загрузится.

## 8. Где что в коде (`index.html`, один файл)

- `SURAHS` — список сур. `ST` — состояние (всё сохраняется в localStorage).
- `renderCB` — верхняя панель; `renderBottomNav` — нижняя навигация;
  `renderViewPanel` — панель ⚙ Вид; `renderReading` / `renderMushafPage` — чтение.
- `mdRender` — Markdown. `PACKS` / `fitMushafLines` — печатный мусхаф.
- `applyView` — CSS-переменные размеров/интервалов/полей
  (`--rs-ar/--rs-ru/--lh-ar/--lh-ru/--mar/--sb`).
